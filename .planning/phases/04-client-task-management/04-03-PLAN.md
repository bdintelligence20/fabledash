  <name>Task 1: Build task list page with multi-filter controls and data table</name>
  <files>src/pages/TasksPage.tsx</files>
  <action>
Replace the placeholder TasksPage with a full task management page:

**Data fetching:**
- Fetch tasks from /tasks endpoint via apiClient.get, passing query params for active filters
- Fetch clients from /clients endpoint to populate client filter dropdown and resolve client names in table
- Build a clientMap (Record of id -> name) from clients response for display
- Re-fetch tasks when any filter changes (status, priority, client_id, assigned_to query params)
- Type responses matching backend models:
  - TaskResponse: id, title, description, client_id, status, priority, due_date, assigned_to, comments[], attachments[], created_at, updated_at, created_by
  - TaskStatus enum values: todo, in_progress, in_review, done, blocked
  - TaskPriority enum values: low, medium, high, urgent

**Layout and controls:**
- Page header: "Tasks" title with "New Task" Button (variant="primary") opening create modal
- Filter bar row with 4 Select dropdowns + 1 search Input, all in a responsive flex-wrap layout:
  - Search Input: filters task titles client-side via .filter() on the fetched list
  - Status Select: options All + 5 statuses (Todo, In Progress, In Review, Done, Blocked) — sends ?status={value} to API
  - Priority Select: options All + 4 priorities (Low, Medium, High, Urgent) — sends ?priority={value} to API
  - Client Select: options All + list of client names from /clients — sends ?client_id={value} to API
  - Clear Filters button (secondary variant) resets all filters

**Summary StatCards row (between filters and table):**
- 4 StatCards in grid-cols-2 md:grid-cols-4:
  - Total Tasks: total count
  - In Progress: count of status=in_progress
  - Overdue: count of tasks where due_date < now and status != done
  - Completed: count of status=done

**Data table:**
- Table compound component with columns:
  - Checkbox (for bulk select) — simple controlled checkboxes, store selected task IDs in state
  - Title (bold, linked to /tasks/{id} for future detail page)
  - Client (resolved name from clientMap, or "Unknown" if client_id not found)
  - Status (Badge with status-specific colors: todo=secondary, in_progress=primary, in_review=warning, done=success, blocked=danger)
  - Priority (Badge with priority colors: low=secondary, medium=primary, high=warning, urgent=danger)
  - Due Date (formatted as "MMM DD, YYYY" or "No date" if null; show in red text if overdue)
  - Actions column: Edit button, Delete button (with confirmation)

**Bulk actions bar:**
- Appears when 1+ tasks are selected (sticky or fixed at bottom of table area)
- Shows "{N} selected" count
- Bulk status change: Select dropdown with status options + "Apply" button
- Applies PUT /tasks/{id} for each selected task with the new status, then refreshes list
- "Clear selection" button

**Create Task Modal:**
- Modal with title "New Task"
- Fields: title (Input, required), description (Input/textarea), client_id (Select, required — populated from client list), priority (Select, default medium), due_date (Input type="date"), assigned_to (Input, optional — free text UID for now)
- Status defaults to "todo" on creation (not shown in form)
- Submit: POST /tasks via apiClient
- On success: close modal, refresh task list
- On error: display error in modal

**Status badge color mapping (reusable const):**
- todo -> "secondary"
- in_progress -> "primary"
- in_review -> "warning"
- done -> "success"
- blocked -> "danger"

**Priority badge color mapping:**
- low -> "secondary"
- medium -> "primary"
- high -> "warning"
- urgent -> "danger"

**Date formatting:**
- Use toLocaleDateString('en-ZA', { day: 'numeric', month: 'short', year: 'numeric' }) for due dates
- Overdue check: new Date(due_date) < new Date() && status !== 'done'

**Important patterns:**
- Relative imports from ../components/ui and ../lib/api
- No external state management — local useState/useEffect
- Filter params sent as query string to backend (backend handles Firestore .where() filtering)
- Client-side search filters the already-fetched list (no API call for search)
