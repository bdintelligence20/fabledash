  <files>src/pages/TaskDetailPage.tsx</files>
  <action>
Replace the placeholder TaskDetailPage with a full task detail view:

**Data fetching:**
- Extract taskId from useParams() (react-router-dom)
- Fetch task data: apiClient.get(`/tasks/${taskId}`) on mount
- Fetch client data: apiClient.get(`/clients/${task.client_id}`) to display client name (after task loads)
- Handle loading (Spinner), error, and 404 states

**Page header section:**
- Back link: "< Back to Tasks" using Link to /tasks
- Task title as page title (text-2xl font-bold), editable on click (inline edit or edit button)
- Status Badge with current status (same color mapping as 04-03)
- Priority Badge
- Action buttons row: Edit (opens edit modal), Delete (with confirm dialog), status transition buttons

**Status transition buttons:**
- Show contextual next-status buttons based on current status:
  - todo -> "Start" (in_progress)
  - in_progress -> "Submit for Review" (in_review), "Block" (blocked)
  - in_review -> "Approve" (done), "Return" (in_progress)
  - blocked -> "Unblock" (todo)
  - done -> "Reopen" (todo)
- Each button calls PUT /tasks/{id} with { status: newStatus } and refreshes
- Use appropriate button variants: primary for forward transitions, secondary for backwards, danger for block

**Task info Card (two-column grid on desktop):**
- Left column — editable fields (via edit modal, not inline):
  - Title, Description (rendered as paragraph, "No description" if empty)
  - Client (show name, linked to /clients/{client_id})
  - Priority (Badge)
  - Due Date (formatted, red if overdue and not done)
  - Assigned To (display UID or "Unassigned")
- Right column — metadata (read-only):
  - Created by, Created at (formatted datetime)
  - Last updated (formatted datetime)
  - Comment count, Attachment count

**Edit Task Modal:**
- Modal with title "Edit Task"
- Fields: title (Input), description (textarea), client_id (Select — needs client list fetch), priority (Select), due_date (Input type="date"), assigned_to (Input)
- Status NOT editable in modal — use status transition buttons instead
- Pre-filled with current values
- Submit: PUT /tasks/{id} with changed fields via apiClient
- On success: close modal, refresh task data

**Delete confirmation:**
- Simple confirm pattern: button click sets a "confirming" state, shows "Are you sure? Confirm / Cancel" inline or via a small Modal
- On confirm: DELETE /tasks/{id}, navigate to /tasks on success
