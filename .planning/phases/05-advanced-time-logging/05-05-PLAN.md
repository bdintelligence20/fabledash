<objective>
Build utilization rate calculation (billable vs total hours) and saturation leaderboards (top 5 tasks/projects consuming the most time) with a dedicated dashboard page.

Purpose: Utilization rate is the single most important operational efficiency metric for a creative agency. It answers "How much of our logged time is actually billable?" The CEO monitors this to detect over-servicing (high hours, low billable ratio) and under-utilization. Saturation leaderboards reveal where effort concentrates — the top 5 tasks/clients consuming the most time — surfacing potential scope creep or resource bottlenecks.
Output: Backend endpoint for utilization and saturation metrics, plus a UtilizationPage at /time/utilization with rate gauges and leaderboard tables.
</objective>

  <name>Task 1: Create utilization rate and saturation leaderboard endpoint</name>
  <files>python-backend/app/api/time_logs.py</files>
  <action>
    Add a new GET endpoint to the existing time_logs router:

    **GET /time-logs/utilization** — Calculate utilization rate and saturation leaderboards:
    - Query params: date_from (date, optional), date_to (date, optional)
    - Logic:
      1. Fetch all time logs within the date range (same Firestore query pattern as list/allocation endpoints)
      2. Calculate utilization metrics:
         - total_logged_hours: sum of all duration_minutes / 60
         - total_billable_hours: sum of duration_minutes where is_billable == True / 60
         - total_non_billable_hours: total - billable
         - utilization_rate: (billable / total) * 100 if total > 0 else 0 (percentage, 1 decimal)
      3. Calculate saturation leaderboards:
         - **By Client** (top 5): group by client_id, sum duration_minutes, sort descending, take top 5. Resolve client names by fetching from clients collection. Include: client_name, total_hours, percentage_of_total, entry_count.
         - **By Task** (top 5): group by task_id (skip entries where task_id is null), sum duration_minutes, sort descending, take top 5. Resolve task names by fetching from tasks collection (COLLECTION_NAME from task model). Include: task_name, client_name (via task's client association or time log's client_id), total_hours, percentage_of_total, entry_count.
      4. Calculate daily trend (for sparkline/chart):
         - Group by date, sum total_hours and billable_hours per day
         - Return as array sorted by date ascending: [{ date, total_hours, billable_hours, utilization_rate }]
    - Response shape:
      ```json
      {
        "success": true,
        "data": {
          "period": { "from": "2026-01-01", "to": "2026-02-25" },
          "utilization": {
            "total_logged_hours": 320.5,
            "total_billable_hours": 256.4,
            "total_non_billable_hours": 64.1,
            "utilization_rate": 80.0
          },
          "saturation_by_client": [
            { "client_id": "c1", "client_name": "Acme Corp", "total_hours": 85.5, "percentage": 26.7, "entry_count": 45 }
          ],
          "saturation_by_task": [
            { "task_id": "t1", "task_name": "Brand Design", "client_name": "Acme Corp", "total_hours": 42.0, "percentage": 13.1, "entry_count": 22 }
          ],
          "daily_trend": [
            { "date": "2026-02-01", "total_hours": 8.0, "billable_hours": 6.5, "utilization_rate": 81.3 }
          ]
        }
      }
      ```

    IMPORTANT: Place this route BEFORE the `/{time_log_id}` GET route (alongside /allocation) to avoid path parameter conflicts.

    Import tasks COLLECTION_NAME: `from app.models.task import COLLECTION_NAME as TASK_COLLECTION` for task name resolution.
